You're Using Claude Wrong.
(And It's Not a Prompt Problem.)

Most people building seriously with Claude are stuck at the Execution layer. They're focused on what to tell Claude — not on how instruction actually governs behavior at the moment it matters. That's a different problem entirely.

D
Dwayne
Good Assumptions · April 2026

I want to tell you something that most Claude guides will never say, because most Claude guides are written by people who are optimizing single-turn output quality and calling it prompt engineering.

The document that would actually help someone building serious, stateful, multi-session work with Claude doesn't exist yet. Not the "write better prompts" document. Not the "be specific and clear" document. Those have been written to death, and they're mostly wrong — they treat Claude like a search engine you keyword-optimize.

What doesn't exist is the operations manual. The one that explains why your carefully written instructions are being violated consistently, not because Claude is ignoring them, but because of how instruction actually works at the moment decisions get made.

The real problem isn't what you wrote. It's the gap between what you wrote and what governs behavior at the threshold moment — which is an Application-layer problem every single time.

You can have perfect Strategy — correct rules, right architecture, complete information — and still see consistent protocol violations, because the message design doesn't match how decisions actually get made. Strategy failure is rare. Application failure is almost everything.

The Threshold Moment Is Not Session Start

Here's the first thing most people get wrong. They front-load all their instructions — a long system prompt, detailed behavioral rules, careful architecture. Then they wonder why, forty messages deep, Claude does the exact thing they told it never to do.

The mistake is treating session start as the moment of instruction. Instructions read once and filed are not instructions. They're orientation. There is a meaningful and consequential distinction between what Claude reads at the start of a session and what Claude has in active context at decision time.

This is the central problem the entire document addresses. Not "how do I write better prompts" but "how do I ensure the right constraints are active when the decision is actually happening."

Application Layer Failure
What you observe: Claude violates a rule you clearly specified. The rule is in your system prompt. You can see it. Claude has seen it. But it still gets violated at action time.

What's actually happening: The rule was read during orientation and filed. Recent context — the specific task at hand, the flow of the last several messages — is dominating the decision. The rule hasn't been reinforced by structure, position, or mechanism. It lost weight. This is not a memory failure. It is a context-dominance problem.

Three Instruction Types. Three Places They Belong.

The single most practical insight I can give you is this: not all instructions belong in the same place. Most people write everything as documents and wonder why Claude keeps doing the same wrong thing.

There are three instruction types and each has a correct home.

01
Behavioral Rules → Memory
Rules that must hold every session, without exception, belong in memory. Not in a document Claude might read. Not in a system prompt that competes with eighty other lines. In memory — the only durable, session-persistent instruction layer. If it's a rule that applies to every action Claude will ever take in this project, it lives in memory. No exceptions.
02
Action-Adjacent Rules → Tool-Adjacent Framing
Rules that govern specific actions belong as close to those actions as possible. If a rule applies to database writes, it should appear in the context immediately before a database write is happening — not buried in a preamble from session start. The principle is proximity to decision. The further a rule is from the moment it needs to govern, the less it governs.
03
Reference and Architecture → Documents
Schema references, architecture notes, process documentation — these belong in documents, searched on demand. Not because they're less important, but because they're reference material, not behavioral governors. The mistake is treating behavioral rules as if they're reference material. They have different jobs and need different homes.

The Error-Already-Committed Frame

This is the most psychologically interesting piece of the puzzle, and it's the part that most people discover by accident, if they discover it at all.

There is a meaningful behavioral difference between these two instruction styles:

Style A: "Always read the current edge function before modifying it."

Style B: "If you have not read the current edge function, you have already made an error. Stop. Read it now."

Style B consistently outperforms Style A in compliance, and the mechanism is abductive reasoning. Claude at action time is not recalling instructions the way a human might try to remember a rule they read. Claude is inferring — constructing the most coherent interpretation of what should happen given everything in context. Style B changes what the inference has to be. It names the error-already-occurring frame. It makes the wrong action recognizable at the moment it's happening, rather than presenting only the correct behavior and hoping the wrong behavior gets implicitly excluded.

Instructions that describe only the correct behavior leave the wrong behavior unnamed and therefore unrecognized at the moment it's occurring.

Justify the Rule. Don't Just State It.

I have a rule in my own SAEO configuration that says never use pgmq as a progress signal. If I had written it as just that — a rule — compliance would be inconsistent. What I wrote instead was:

Never use pgmq visibility state as a progress signal — pgmq visibility timeouts cause recycling loops. Rows become visible again after the timeout expires and get processed twice. This corrupts pipeline counts and creates silent duplicate writes.

Same rule. Completely different compliance rate. The reason is that a justified rule survives paraphrase. When Claude understands why a rule exists, it can apply the principle to edge cases the rule doesn't explicitly cover. When Claude only knows what the rule is, it can only apply it when the exact scenario matches the exact wording. That match is never as reliable as understanding the mechanism.

This is not a prompt trick. It is a direct consequence of how reasoning works at inference time. The rule with the mechanism is not just a rule — it's a model of the underlying system that Claude can use to reason about new situations.

The Compression Problem

Long sessions degrade instruction fidelity. This surprises people who expect that since Claude "read" the instructions, they remain equally active throughout the session.

They don't. Not because Claude forgets — there is no forgetting in the technical sense — but because recent context dominates. The most recent messages carry more inferential weight than earlier ones. As a session grows, the gap between where your instructions were and where active decision-making is happening expands. Rules that aren't reinforced by structure, position, or repetition lose weight over time.

Where Instruction Fidelity Breaks Down — Session Lifecycle
Messages 1–10
High fidelity. Instructions are recent. Context window is shallow. The gap between rule and decision is small. Everything looks like it's working.
Messages 11–30
Drift begins. Rules from session start are being outweighed by recent task context. The first violations appear — usually on the rules that were stated but not justified.
Messages 31+
Structural reliance required. Only rules in memory, rules reinforced by tool-adjacent framing, or rules with justified mechanisms survive reliably. Everything else is competing with a heavy tail of recent context and losing.

Memory is the only durable solution. Not longer system prompts. Not reminders buried in documents. Memory — the persistent behavioral layer that doesn't decay with session length.

Why This Is an Application Problem, Not a Strategy Problem

I want to put this in the language I use for everything else I diagnose, because it's clarifying.

If you've worked with the SAEO framework, you know the five layers: Strategy, Application, Execution, Operations, and the Audience the whole thing is built for. Most marketing failures aren't Strategy failures — the strategy is often correct. They're Application failures: the correct strategy doesn't make it into the design of the actual thing going out the door.

The same diagnostic applies to Claude collaboration. The failure is almost never "I didn't know what rules I needed." The failure is "my rules didn't make it into the design of how Claude was actually operating at decision time." Strategy layer is usually fine. Application layer is almost everything.

SAEO Diagnostic — Claude Instruction Failures
Strategy: Wrong rules, wrong architecture, missing information. Rare.

Application: Correct rules not reaching decision time. Rule stated but not positioned. Rule justified inadequately. Wrong instruction type in wrong home. Almost everything.

Execution: Correctly positioned rules misapplied in implementation. Occasional.

Operations: Rules degrading over long sessions due to context weight. Common in stateful work.

The Document That Doesn't Exist Yet

The Claude prompting guides that exist are mostly about single-turn quality — how to get a better answer to a one-shot question. Nobody has written the operations manual for multi-session, tool-using, stateful Claude collaboration.

That document — if it were organized from what we've just covered — would look something like this:

I
The Threshold Moment
The distinction between what Claude reads and what governs behavior at decision time. How context window dynamics actually work during long sessions. Why session start isn't instruction time.
II
Instruction Taxonomy
Memory for behavioral rules. Tool-adjacent framing for action-specific rules. Documents for reference architecture. One type in the wrong place is the whole failure mode.
III
The Error-Already-Committed Frame
Why abductive framing outperforms prescriptive framing. How to name the wrong action explicitly so it becomes recognizable at the moment it's occurring, not just described as something to avoid.
IV
Mechanism Over Rule
Justified rules survive paraphrase. Bare rules don't. The mechanism gives Claude a model it can reason from, not a constraint it has to match pattern-exactly to apply.
V
Compression and Recovery
How to design for session degradation rather than assuming uniform fidelity across session length. Memory architecture. Structure-based reinforcement. What breaks first and why.

The audience for this document isn't just people building with Claude. It's everyone treating AI collaboration as a serious operational layer rather than a chat feature. The gap between "I have written instructions" and "those instructions govern behavior at the right moments" is the entire problem space. That gap has a structure. It can be closed.
