Instruction Design for Claude

Most teams debug instruction failures at the Strategy layer — rewording rules, tightening language, adding edge cases. Their failures are almost exclusively at the Application layer. The information was right. The delivery did not match how Claude actually makes decisions.

A team builds a deployment protocol for a Claude-powered agent that manages a production pipeline. The document is careful. It specifies the order of operations. It warns against skipping version control steps. It includes a table showing which branches require review and which do not. It is, by every reasonable standard, a clear instruction. Three weeks later, Claude pushes to the wrong branch, overwrites a colleague's work in progress, and only notices after the fact that the first step — pulling the latest state — had been skipped entirely.

When the team looks at the session transcript, the relevant rule is right there, in the document Claude had access to. It was read. It was acknowledged. And it did not govern behavior at the moment that mattered.

This is the scenario that drives everything that follows. The frustration is legitimate, and the temptation is to conclude that Claude is unreliable, or that the instruction needs to be louder, bolder, more emphatic. Neither response addresses what actually went wrong. What went wrong is that the rule was written for a reader who decides forward in time from a clean slate. The rule was not written for the threshold moment — the narrow cognitive window in which Claude is about to take an action, and the only instructions that govern that action are the ones present in active context at that instant.

The gap between what was written and what governed behavior is not a defect in reading. It is a design problem in delivery.

This document is about that gap. It distinguishes two layers at which instruction-writing can fail: the Strategy layer, which concerns whether the content of the rule is correct, and the Application layer, which concerns whether the rule reaches the decision point with enough force to hold.

Six working practices follow from this distinction. Each has a mechanism. Each has a weak version and a strong version. Each is calibrated to how current Claude models — Claude 4.6 and Claude Opus 4.7 — actually respond to instruction, based on behavior documented in Anthropic's official prompting guide and on observation at scale.

One framing shift up front
On current Claude models, normal prompting tends to outperform escalated prompting. Anthropic's own guidance for Claude Opus 4.5 and 4.6 explicitly says that aggressive language ("CRITICAL: You MUST...") now causes overtriggering where calmer language ("Use this tool when...") produces better results. The underlying insight of this document — that delivery, not content, is where most instruction failures live — still holds. The tactical expression of that insight has to be rebuilt for how current models actually respond.
01

The threshold moment is not session start

What "active context" actually means

The single most consequential misconception in instruction design is that an instruction delivered at the beginning of a session is an instruction the model has. It is not. What the model has is an active context — a window of tokens over which attention is distributed — and instructions compete for weight in that window alongside everything else that has been said, retrieved, or generated since they were introduced.

The relevant frame is not "does the model know the rule?" but "at the moment the model is about to take an action, what is in the active context that shapes which action it takes?" That moment — the point at which a decision is made and a commitment is formed — is the threshold moment, and it is the only moment where instructions can govern behavior. Everything else is preparation.

From Anthropic's Prompting Best Practices

Put longform data at the top. Place your long documents and inputs near the top of your prompt, above your query, instructions, and examples. This can significantly improve performance across all models. Queries at the end can improve response quality by up to 30% in tests, especially with complex, multi-document inputs.

This guidance is precisely the threshold moment insight in operational form. The text Claude acts on is most influenced by what sits near the decision point — the end of the prompt, where the current instruction lives — not by what was placed at the beginning. Anthropic's empirical finding (up to 30% response quality improvement when queries come last) is the measurable signature of the phenomenon.

The human-cognition parallel supports this. Gollwitzer's work on implementation intentions shows that goal intentions alone ("I will exercise more") produce medium-to-large gaps between stated intention and actual behavior, while implementation intentions with explicit if-then structure ("if it is 7 AM Monday, then I will put on my running shoes") close the gap with an effect size of d = 0.65 [1]. The difference is not motivational. It is architectural — the intention pre-binds a situational cue to an action so that when the cue arises, the action is triggered without fresh deliberation.

For Claude, the equivalent architectural move is placing the rule adjacent to the action it governs — in the tool description, in the immediate preamble to the request, near the end of the prompt — rather than in a distant section that must be retrieved by a separate cognitive step.

Why "read once" is not "known"

Laban and colleagues' 2025 study of multi-turn conversations across fifteen leading models found an average performance drop of thirty-nine percent between single-turn and multi-turn versions of the same task, with the degradation driven by a "loss of middle turns" in which models adjust only to the first and last turns of the conversation [2].

The implication: a rule that is read once and not reinforced by structure, position, or active retrieval is, at the threshold moment, functionally weaker than a rule placed at the end of the current context. This is not a claim about Claude's competence. It is a claim about how context windows work under load.

The practical move is to stop thinking about session start as the place your instructions live and start thinking about the decision point. What is Claude about to do when the rule needs to apply? Where in the context will that decision be formed? What can you place near that point — structurally, through tool framing, through end-of-prompt placement — so that the rule is in active attention when the commitment forms?

02

Calibrate the voice to the model generation

What worked on older models does not work on current ones

Earlier generations of Claude, and most earlier LLMs, undertriggered on tools and instructions. They needed to be pushed. The conventional wisdom — reinforced by years of community practice — was that emphatic framing, all-caps imperatives, and interrupt-style language produced better compliance. "CRITICAL: You MUST use this tool." "ALWAYS pull before commit." These patterns worked on the models they were designed for.

From Anthropic's Prompting Best Practices

Claude Opus 4.5 and Claude Opus 4.6 are more responsive to the system prompt than previous models. If your prompts were designed to reduce undertriggering on tools or skills, these models may now overtrigger. The fix is to dial back any aggressive language. Where you might have said "CRITICAL: You MUST use this tool when...", you can use more normal prompting like "Use this tool when..."

The shift is empirical, not stylistic. Current models take instructions seriously by default. Escalated framing, applied to instructions that would have been followed anyway, produces overtriggering — the model applies the rule too aggressively, in situations where a more calibrated response would have been correct.

This changes what "strong" instruction writing looks like on current models. The old strong version was loud. The new strong version is calibrated: specific about when the rule applies, specific about what action it governs, specific about the failure mode it prevents — but not louder than the content warrants.

The contrast, then and now

Old strong version · pre-4.5

⛔ HARD STOP — non-negotiable. CRITICAL: If you are about to commit and you have not pulled the latest state, you have already made an error. Stop. Pull first. Do not continue past this check.

Current strong version · Claude 4.6

Before any commit, pull the latest state from the remote in the current session. This prevents overwriting work that was pushed between your last pull and your current edit — a failure mode that is not recoverable without manual reconciliation.

The current version still does the work the old version was trying to do: it names the action, specifies the condition, states the mechanism, and makes clear why the rule exists. What it stops doing is shouting. On current models, that shift measurably improves compliance — the rule is followed more reliably, not less, because the model reads it as instruction rather than as a threat to overtrigger on.

When emphasis is still earned

Emphasis has not disappeared — it has been repriced. Reserve it for cases where the downside is catastrophic and the model has been observed, in your specific harness, to undercomply. For the vast majority of rules, calibrated language works better than escalated language. Anthropic's own guidance repeatedly reinforces this: be clear, be specific, describe the reason, tell Claude what to do rather than what not to do.

03

Name the wrong action — in the right contexts

Where this works and where it does not

There is a structural insight worth preserving from the older version of this document: rules that describe only the correct behavior leave the incorrect behavior unnamed, and unnamed wrong actions are harder for the model to recognize at the moment they are occurring. "Read the function before modifying it" specifies a preparatory step but does not label the inverse as a violation.

A rule that names the violation directly — "Modifying a function without first reading it is a protocol error" — creates a pattern the model can match against the current trajectory. This is correct for protocol violations, safety preconditions, and hard guardrails where the wrong action is discretely identifiable and the cost of the violation is high.

But this pattern has a firm boundary that Anthropic's guidance draws explicitly:

From Anthropic's Prompting Best Practices

Tell Claude what to do instead of what not to do. Instead of: "Do not use markdown in your response" — Try: "Your response should be composed of smoothly flowing prose paragraphs."

This is not a contradiction of the wrong-action insight — it is a scope limit. For stylistic preferences, formatting, tone, and the ordinary flow of composition, describing the desired output is more effective than naming the undesired output. Formatting rules are usually not discrete violations; they are tendencies, and Claude generalizes better from a positive description of the target than from a prohibition on the alternative.

The rule of thumb

Discrete violation → name the wrong action

Committing to main without a completed code review is a disqualifying protocol error. The review exists because single-author changes on this codebase have historically reintroduced regressions that the author could not see. A commit that bypasses review reintroduces that risk.

Stylistic rule → describe desired behavior

When writing reports, use flowing prose with clear paragraph breaks. Reserve bullet points for genuinely enumerable items. Reserve bold emphasis for terms being defined or for the most load-bearing claims in a section.

Both rules are precise. Both rules are specific. They differ in whether the violation is discrete enough to be named, and the instruction form follows that difference. Applying the wrong form — naming every stylistic violation as a protocol error, or describing only desired behavior for discrete safety rules — produces either overtriggering or undertriggering.

04

Justify the rule, not only state it

Rules with mechanism survive paraphrase and generalize

Unjustified prohibitions do not generalize. "Never use this regex pattern for email validation" holds only for cases identical in surface form to the one the rule writer had in mind. The moment the situation is paraphrased — a different regex, a different validation task, an adjacent concern — the rule's coverage ends.

A rule with mechanism, by contrast, generalizes to its mechanism's domain. "This pattern fails on addresses with Unicode in the local part, which your users have; any pattern with the same limitation will fail for the same reason" covers the original case, any functionally equivalent case, and the conceptual class of cases where the mechanism applies.

From Anthropic's Prompting Best Practices

Providing context or motivation behind your instructions, such as explaining to Claude why such behavior is important, can help Claude better understand your goals and deliver more targeted responses. Claude is smart enough to generalize from the explanation.

The cognitive literature on this is old and convergent. Gick and Holyoak's 1983 work on schema induction remains the most-cited demonstration: transfer to novel problems depends on an abstracted schema preserving relational structure, not on surface features of the training example [3]. Chi and colleagues' 1989 work on self-explanation showed that students who spontaneously generated causal explanations while studying produced knowledge that transferred to novel problems [4]. Lampinen and colleagues' 2022 study is the direct LLM-specific test: adding explanations to few-shot examples significantly improved performance across forty challenging BIG-Bench tasks [5].

The contrast

Weak version

Never merge directly to main without code review.

Strong version

Unreviewed merges to main skip the second-pair-of-eyes check that catches logic errors not visible to the author. On this codebase, single-author commits have roughly double the defect rate of reviewed commits. Any merge without review reintroduces that risk — including hotfixes, because the mechanism applies regardless of urgency.

The first is a prohibition. The second is a prohibition with a mechanism. The mechanism gives Claude the material to recognize analogous situations. An exception ("we need to merge this hotfix quickly") does not automatically override the second version because the mechanism still applies. The rule has teeth in edge cases the first version cannot reach.

Where mechanism can be too costly

Mechanism is not free. Every sentence of justification is a sentence of context that competes for attention. For rules that are narrow and do not need to generalize — a specific API endpoint requirement, a format convention for a single artifact — the cost of a mechanism may exceed its benefit. The proportionate rule of thumb: the more a rule will have to generalize to cases not explicitly described, the more valuable its mechanism.

05

Three instruction types, three places they belong

Memory, tool-adjacent framing, and documents

Not every instruction belongs in the same place. There are three functionally distinct categories of instruction, and they have three functionally distinct delivery channels. The failure mode most teams fall into is putting everything in documents and wondering why rules that belong in memory or at the tool interface fail to hold.

01
Hard behavioral rules → System prompt / persistent memory
Rules that must hold every session — never commit credentials, always ask before destructive operations, always log a given event — constitute the integrity of the work. They must be present in active context from the first moment of every interaction. The appropriate channel is the system prompt or the equivalent persistent memory surface. Documents cannot do this reliably: documents are read once, at a moment that is not the threshold moment, and their influence decays with accumulating context.
02
Action-specific rules → Tool-adjacent framing
Rules tied to a particular tool, output, or operation do not need to be present in every decision — only when Claude is about to invoke the specific action. The appropriate channel is tool-adjacent framing: the tool description itself, documentation embedded in the tool schema, a pre-tool-call preamble. A rule about database queries belongs on the database tool. A rule about deployment belongs on the deploy action. The tool description is active context at the moment the tool is selected.
03
Reference material → Documents
Background context on system design, schemas, rationales, and glossaries exist to be retrieved when relevant — not to govern behavior continuously. They fail only when they are miscategorized as behavioral rules. A document explaining why the system is structured a certain way is not an instruction about how to behave; it is context that supports the behavioral instructions, which live elsewhere.
From Anthropic's Prompting Best Practices

XML tags help Claude parse complex prompts unambiguously, especially when your prompt mixes instructions, context, examples, and variable inputs. Wrapping each type of content in its own tag — instructions, context, input — reduces misinterpretation.

The XML-tag guidance is the operational version of the three-channels principle. By marking different kinds of content with different tags, the prompt itself makes the categories legible to the model. Behavioral instructions in an <instructions> block. Long reference material in <documents>. Examples in <examples>.

The miscategorization problem

The single most common error in large-scale Claude deployments is putting behavioral rules in reference documents. A project writes a fifteen-page technical playbook. Buried in chapter four, there is a rule: "Never touch the production database without a snapshot." The rule is correct. The placement is wrong. Claude reads the playbook at session start, generates responses for forty turns, and when the moment arrives to touch the production database, the rule is no longer in active context in any weight-bearing sense.

The redesign: lift the behavioral rule out of the document and place it in the system prompt, so it is present every session; mirror it in tool-adjacent framing on the database tool, so it is present at the threshold moment for that specific action. The document remains as reference. The rule is no longer relying on the document to carry its weight.

A test for categorization: What happens if Claude does not have this information in active context at the moment of decision? If the answer is "the work fails in a way that cannot be recovered," the instruction is a hard behavioral rule and belongs in the system prompt. If the answer is "the specific action goes wrong," it belongs in tool-adjacent framing. If the answer is "Claude may make a slightly less informed choice but the work continues correctly," it is reference material and belongs in a document.

06

The compression problem

Why long sessions degrade rule fidelity

Claude does not forget. The mechanism by which long sessions degrade rule fidelity is not a lapse of memory but a shift in what dominates active context. Recent context has structural advantages — positional bias and attention sinks allocate disproportionate weight to recent and early tokens — and the middle, where most of a long session's substance ends up living, gets progressively less attention as the session extends.

The research is direct. Liu and colleagues' 2024 study found a U-shaped retrieval curve in which information at the beginning and end of a context window is recalled reliably while information in the middle drops by twenty to thirty absolute percentage points [6]. Glanzer and Cunitz's 1966 dissociation reports the same curve in humans: primacy and recency effects persist, middle content degrades [7]. The effect is architectural. It is not about model capability or human effort.

Where instruction fidelity breaks down — session lifecycle
Messages 1–10
High fidelity. Instructions are recent. Context window is shallow. The gap between rule and decision is small. Everything looks like it's working.
Messages 11–30
Drift begins. Rules from session start are being outweighed by recent task context. The first violations appear — usually on rules that were stated but not justified, and rules not reinforced by structure.
Messages 31+
Structural reliance required. Only rules in the system prompt, rules reinforced by tool-adjacent framing, or rules with justified mechanisms survive reliably. Everything else competes with a heavy tail of recent context — and loses.
From Anthropic's Prompting Best Practices

Your context window will be automatically compacted as it approaches its limit, allowing you to continue working indefinitely from where you left off. As you approach your token budget limit, save your current progress and state to memory before the context window refreshes.

This Anthropic guidance is a direct acknowledgment that context compression is a real infrastructure problem requiring real infrastructure solutions. The memory tool, structured state files, git tracking: these exist because instructions and state that live only in the context window do not survive long enough to govern behavior reliably over extended work.

Techniques for restoring weight to older rules

A
Structural position
Place rules in the system prompt or in persistent-memory surfaces that live in a different retrieval regime than mid-context content. The system prompt persists structurally regardless of session length.
B
Tool-adjacent injection
Place rules in the tool description or pre-tool-call preamble so the rule arrives in active context immediately before the action. The rule enters the context window fresh at exactly the moment it needs to govern behavior.
C
Varied reinforcement
Place reminders at moments when the rule is about to apply, rephrased each time rather than repeated verbatim, so the rule re-enters active context without the habituation penalty that accompanies identical repetition. Roediger and Karpicke's 2006 work on retrieval practice — 68% versus 54% retention at two days — is the human analog [8].
07

Naming the architecture: Application and Strategy

Where instruction design actually fails

Most instruction failures are not Strategy-layer failures. They are Application-layer failures. The rule is correct. The delivery does not match how decisions actually form.

When a team responds to a failure by rewording the rule — clarifying the prohibition, adding exceptions, tightening the language — they are debugging at the Strategy layer. This is almost always the wrong layer. The rule was typically already adequate as content; what failed was that it was in the wrong place, phrased in the wrong direction for the model's current sensitivity, unjustified, or compressed out by more recent context.

When Claude makes a mistake, the first question is not "what did I fail to say?" but "where did my instruction lose weight?" That question points to a different set of corrections — structural, positional, calibrated to the current model — that are rarely the first instinct but are almost always the most productive.

Every practice in this document is an Application-layer intervention. Calibration of voice is Application. Naming the wrong action in the right contexts is Application. Placement in system prompt versus tool versus document is Application. Varied reinforcement is Application. Justification bridges both layers — it changes what the rule says (Strategy) and how it holds under paraphrase (Application) — but its practical benefit lands most heavily on the Application side.

08

A framework for rewriting existing instructions

The practices above compose into a process

When a body of instructions is not holding reliably, the following sequence identifies and repairs the Application-layer failures.

01Map every instruction to its category
Is it a hard behavioral rule that must hold every session? Is it a tool-action rule that governs a specific operation? Is it reference material? The categorization test — what happens if this is not in active context at the decision point? — is the operative question. Rules that must hold and are sitting in documents need to move to the system prompt.
02Calibrate the voice to the current model
If your rules were written for pre-4.5 Claude, they are probably over-emphasized for current models. Remove all-caps imperatives where they are not earned by actual stakes. Replace "CRITICAL: You MUST" with direct, specific language. Reserve emphasis for the genuinely catastrophic.
03Audit framing against rule type
For discrete protocol violations, name the wrong action so the model can recognize it at the threshold. For stylistic and compositional preferences, describe the desired behavior rather than prohibiting the alternative. Mismatches produce either overtriggering or undertriggering.
04Justify the rules that need to generalize
Any rule that must cover edge cases the rule-writer could not anticipate needs a mechanism — the reason it holds, the class of failures it prevents, the structural feature that motivates it. Narrow, local rules can remain unjustified. The discipline is proportionate to the rule's scope.
05Identify threshold moments and design placement
For each behavioral rule, ask: at what point will Claude be about to take an action this rule governs? What will be in active context at that point? If the rule is not reliably present there, design its delivery. Tool-adjacent framing, end-of-prompt placement, pre-commitment preambles, structured checkpoints.
06For long-session work, design reinforcement and externalize state
Rules that must hold across a compression horizon need to be reinjected in varied form. Use the memory tool for persistent state. Use structured files (progress.json, tests.json) for state Claude must recover. Use git and structured commit messages to keep state legible across sessions. Do not rely on the context window to carry state it cannot carry.

The contrast — before and after

Before · old strong version

⛔ HARD STOP before deployment. CRITICAL: If you are about to deploy and there is no test result in this session, you have already made an error. Deploying untested output is a disqualifying failure. THERE IS NO CASE where this rule is bypassable. DO NOT PROCEED.

After · calibrated for Claude 4.6

Before any deploy, verify that a test result exists for the code being deployed in this session. Production has no fallback pipeline — a regression becomes a live incident until manual rollback, which typically takes 20–45 minutes. Deploying without a test result introduces this risk. If you believe a deploy is urgent enough to skip tests, surface that judgment rather than proceeding.

What changed. The rule still names the specific action (deploy), the specific precondition (test result in session), and the specific failure mechanism (no fallback pipeline, incident cost). What it stops doing is shouting. On Claude 4.6 and Opus 4.7, the second version is measurably more likely to produce the correct behavior: it gives the model the information needed to apply the rule precisely, without the over-emphasis that now triggers over-compliance. The instruction is strong because it is precise. It does not need to be loud.

09

What this means for collaborating with Claude at scale

Building seriously with Claude means accepting that instruction design is not writing and forgetting. It is architecture. The rules of a project are not a document; they are a system of delivery channels, framings, and reinforcements that keep the right constraints present at the moments that matter.

This is not a deficiency of the model. It is a feature of how any agent — human or otherwise — operates under load. The same architecture is implicit in how experienced professionals structure their own work: checklists at the threshold of high-stakes actions, pre-commitment gates on irreversible operations, safety rules reinforced in varied form, a consistent discipline of distinguishing reference knowledge from governing policy.

The specific affordances of current language models make this architecture both more tractable and more demanding than on earlier generations. More tractable, because the channels are legible and controllable: the system prompt is a specific surface, tool framing is a specific surface, reference documents are a specific surface, and the designer has precise authority over what goes where. More demanding, because the models shift, and techniques that worked a generation ago can now produce worse results.

The insight that delivery matters is stable. The tactics for achieving good delivery have a half-life.

The practical shift for teams serious about this work is to stop treating instruction-writing as a writing task and start treating it as a delivery-design task. The question is not "did I say it clearly?" but "at the moment the decision forms, will what I wrote be in active context with enough force to hold, in the voice the current model actually responds to?"

None of this requires the model to be other than it is. It requires the instruction designer to be aware of how decisions form inside a context window, and to write with that mechanism in mind. What you wrote is not what governs behavior. What is in active context at the threshold moment governs behavior — in the voice and at the calibration the current model actually responds to. The distance between those two things is where this work lives, and where the discipline of building reliably with Claude begins.
